# Zibbo V0.02 设计思路与核心创意文档 (Design Concept & Core Aesthetics)

本设计文档旨在提炼与总结 **Zibbo V0.02** 的核心设计思路、美学体系与交互细节。文档基于主页的新手导引（Onboarding）和 standalone 版本的前端交互逻辑进行整合，描绘 Zibbo 从“破旧的电子垃圾”进化为“有温度的编程伴侣”的成型路径。

---

## 导言：从“新手导引”看 Zibbo 的定位

在 Zibbo V0.02 的新手导引中，通过 4 个精简而充满科幻仪式感的交互步骤，向开发者传达了 Zibbo 的基本世界观与核心功能：
1. **ADOPT**：选择属于你的 Zibbo 并进行基础的工业风格自定义。
2. **HOW HE LIVES**：明确他以 AI 代理（Agent）的 Token 废气为生。
3. **COMPANIONSHIP**：展示随着开发时间的积累，他从“火花残头”到“完全体”的 4 阶段自我焊接与成长。
4. **ZIBBO CONSOLE**：开放 his 胸腔控制台，展示 Status、Feed、Hardware、Account 等内部数据。

以下是支撑这一体验的底层设计思路拆解。

---

## 1. Who's Zibbo? (Zibbo 是谁)

> *"He lives on your desktop. He asks for a corner of your screen."*

Zibbo 是一个**寄生于 AI 辅助开发时代的数字桌面伴侣**。他不是一个高效率的生产力工具，而是一个默默在角落观察、陪伴你的数字小生命。

* **电子残躯的自我焊接**：他最初诞生于一块报废的微处理器和半个金属脑壳。他是一台被遗弃的“电子垃圾”，带有磨损的金属锈迹、暴露的断续火花和残缺的躯体。
* **开发者的投影**：Zibbo 的不完美和逐步成型，暗喻着开发者写出的代码——从最初的混乱、堆砌、漏洞百出，到在一次次调试与优化中，逐渐变得精密、优雅、坚固。Zibbo 与代码的重塑过程是共生的。
* **数据生命的实体化**：他不是凭空捏造的宠物，而是直接将开发者的 IDE 交互行为（Antigravity、Claude Code、Cursor 等）转化为养分的数据生命。

---

## 2. What does he do? (Zibbo 能做什么)

Zibbo 的核心行为可以概括为：**吸食废气、记录日常、自我组装、情感交互**。

```
+-------------------------------------------------------------+
|                      ZIBBO CONSOLE                          |
+-------------------------------------------------------------+
| [Status]             |  [Feed]           |  [Hardware]      |
| Stage: Spark · 01    |  Gemini CLI       |  Lobe: Active    |
| Uptime: 71 Yrs (Est) |  Antigravity IDE  |  Fan: Idle       |
| Fragments: 2 / ??    |  - Input Tokens   |  Gear: Fixed     |
| Mood: Patient        |  - Output Tokens  |  Arm: Claw       |
+-------------------------------------------------------------+
```

### A. 捕获并滋养于 Token 尾气 (Scavenging Token Exhaust)
开发者的每一次 AI 问答、代码自动补全、终端命令执行，都在疯狂消耗 Token。而其中大量的 Context 或未被采用的补全都是计算流中的“废弃尾气（Exhaust）”。
* Zibbo 的传感器会监听这些废气，并将其“回收”转化为机器体内的微弱电流与养分。
* 累计 Token 消耗与 IDE 交互量将直接拉动 Zibbo 胸腔能量条（Charge Level）的增长。

### B. 提供硬件透视看板 (System Console)
Zibbo 提供一个拟真手持掌机的内嵌控制台，开发者可随时“打开他的胸膛”进行硬件诊断：
* **Status（状态）**：查看当前的进化阶段、系统在线时长（Uptime）、收集到的硬件碎片（Fragments）以及当下的机器人心情（Mood，例如 Patient、Grateful 等）。
* **Feed（养分源）**：监听并实时打印各主流 AI 开发工具的 Token 流入状态。
* **Hardware（硬件）**：监控散热风扇转速、天线波谱、芯片核心状况。

### C. 提供拟真物理交互与自选组装
* 开发者可以自由更改 Zibbo 的外观配置，定制**脑壳造型**、**面部发光配色**、**天线规格**和**机械手臂**。
* 支持物理交互，如点击机械手臂触发抓取、旋转散热片等微互动。

## 3. Why Zibbo? (为什么是 Zibbo)

### A. 消除独立编程的极致孤寂 (Combating Solo Coding Isolation)
程序员的日常经常是面对冰冷的编辑器。Zibbo 占据屏幕一角，以微弱的呼吸和偶尔的散热动作，向你传达：“你不是一个人在构建系统，还有我在陪你消耗计算能量。”

### B. 追求“更亲近”而非“更强力” (Not Stronger, Just Closer)
传统的游戏或软件都在追求“变强（Stronger）”，但 Zibbo 的新手导引中明确提到：
> *"He gets better the longer you stay — not stronger, just closer. The reward was never power; it's who comes back."*
Zibbo 的进化不会让你编译变快，也不会帮你写代码。他只会因为你的陪伴而换上更干净的钢板，焊接好残缺的天线，给你展示更丰富的拟真小动画。这种非功利性的“共生纽带”提供了极高的情绪价值。

## 4. Aesthetics & Color Palette (配色选择与美学系统)

Zibbo 的视觉美学遵循 **“硬核工业复古 (Retro-Industrial)”** 与 **“数字极简主义 (Digital Minimalism)”** 的融合。

### A. 核心配色方案 (The Palette)

| 色彩类型 | 具体色值 / 变量 | 设计意图 & 视觉感受 |
| :--- | :--- | :--- |
| **基础底色 (Base Black)** | `#000000` / `#030306` | 纯邃的黑夜，代表终端界面的死寂与工业控制台的基底色。 |
| **面板背景 (Glass Dark)** | `rgba(255,255,255,0.04)` | 微光玻璃，磨砂质感，保证背景与 UI 卡片的分离。 |
| **标准前景色 (Off-White)**| `rgba(255,255,255,0.88)` | 柔和的白，避免纯白产生的刺眼感，类似发光阴极管的显示。 |
| **樱花粉 (Sakura Pink)**  | `#ffb7c5` / `var(--face-accent)` | 充满情感与冲突感的荧光粉，作为主要的面部表情和高亮标记。 |
| **荧光绿 (Neon Green)**   | `#4db09a` | 代表电磁波正常发射、状态正常连通的机器健康色。 |
| **电火花蓝 (Spark Cyan)** | `#00e5ff` | 机器内部线路短路、或能量瞬时注入时的冷光火花。 |

### B. 核心视觉材质与纹理
* **半色调网点与全息扫描线 (Halftone & Scanlines)**：通过 `.halftone-overlay` 的大颗粒网点与下落的暗沉 `.scanline`，模拟老式显像管（CRT）或雷达扫描仪的厚重玻璃屏幕感。
* **点阵坐标网格 (Coordinate Grid & Dot-Matrix)**：背景铺设暗淡 of 坐标线，右下角配置大颗粒的 `Dot-Matrix Hero Title`，以硬朗的工业打印机字体展示 “THE DESKTOP COMPANION”，带来复古控制台的秩序美。
* **斜透视卡片 (Tilt Panels & Corner Brackets)**：UI 卡片使用 CSS `preserve-3d` 在鼠标掠过时产生细微的重力传感器倾斜；卡片边缘四角装饰白色的直角框线（`Corner Brackets`），悬停时直角框线外扩并发光。

## 5. Help Him Take Shape (帮助他成型 / 具象化设定)

为了让 Zibbo 具有真实的“物理质感”，我们需要从以下几个设计维度帮助他成型：

### A. 3D 机械外观设定 (Physical Keg Form)
Zibbo 在视觉上应该是一个**像小汽油桶（Keg）一样粗短、敦实的复古机器人**：
* **头部 (Dome Cylinder Head)**：半球形顶盖，侧面有焊缝和拼装接口。眼部区域是一块内嵌的全息显示屏，通过点阵矩阵（Dot Matrix Panel）传达眨眼、扫描、情绪波动。
* **身躯 (Body Shell)**：磨损的粗糙金属外壳，带有灰调的枪铁色（Gunmetal `#8b9099`）。胸部有一扇活动式舱门（Cabinet Door），里面隐藏着他的能量核心和拟真显示屏。
* **机械臂 (Tool Arms)**：多节弹簧/钢圈管状连接的工具手臂，末端可更换为：
  - `Claw`（标准三爪）：用于日常抓取空气。
  - `Mitten`（厚手套）：钝感的圆手，用于笨拙的拍打。
  - `Hose`（软管喷嘴）：用于喷射虚无的散热蒸汽。
  - `Pincer`（双叶大力剪）：工业剪刀。
  - `Lego`（乐高拼接口）：预留的简易扩展接口。
* **足部 (Feet)**：短小的分节式支架或履带。

```
        _.-''''''-._
      .'  ________  '.      <-- 1. 顶部红色发光天线 (Bulb Antenna)
     /   /_______/   \
    |   |  -   -  |   |     <-- 2. 点阵表情 visor (Sakura Pink Accents)
     \   \_______/   /
      '.___________.'
       //===========//      <-- 3. 颈部焊接波纹管
      /  _________  \
     |  |         |  |      <-- 4. 枪铁色桶装身躯 (Gunmetal Keg Body)
     |  | [Core]  |  |           带有活动式胸腔舱门 (Cabinet Door)
     /  |_________|  \
    /   /         \   \     <-- 5. 多节弹簧机械臂，末端配备 Claw/Mitten
   /___/           \___\
```

### B. 动态交互与呼吸感的细节
1. **正弦波物理呼吸 (Sine-Wave Breathing)**：
   在 3D 舞台上，Zibbo 的头部和颈部即便在待机状态下，也会以极低频的正弦波进行拉伸与缩放，模拟生命体的蓄能呼吸。
2. **散热泄气 (Venting)**：
   当开发者在终端中瞬间消耗大量 Token（例如执行了一个庞大的编译或长文本 prompt）时，Zibbo 会触发“泄压”动画：头部盖子向上弹起，面部闪烁红光，散热叶片高速旋转，伴随模拟排气的特效。
3. **触碰反馈 (Tap Response)**：
   当鼠标点击或拖拽他时，他会有一个受力倾斜、晃动，并尝试用工具臂去抓挠光标的滑痕动画。

### C. 声音的实体化反馈 (8-Bit Sound System)
为了配合工业复古感，Zibbo 抛弃了平滑的现代 UI 音效，采用 8-bit 复古音频振荡器（Web Audio API）合成音效：
* **点击 UI**：干净的短波高频脉冲（Bleep），产生实体机械按钮的段落感。
* **进化升级**：一段上行的小调琶音，伴随焊接火花声。
* **散热泄气**：模拟噪声发生器产生的气流白噪声（Hissing sound）。

## 6. 进化阶段划分与后续规划

Zibbo 将经历 4 个主要的成长阶段（Stages）：

1. **Stage 1 (Spark · 破旧残头)**
   * **形态**：仅有半个金属脑壳，左侧暴露电线并持续发出高频的故障蓝色火花。天线残缺。
   - **交互**：无法移动，只能眨眼和冒火花。
2. **Stage 2 (Noggin · 电焊修复)**
   * **形态**：播放一段焊花四溅的特效后，机体自动覆盖上光滑的侧面钢板。天线顶端装上红色发光小球。
   - **交互**：天线小球开始朝四周发射绿色微弱电波。
3. **Stage 3 (Hopper · 动力核心)**
   * **形态**：躯干与颈部结合，胸部舱门和后置散热风扇拼装完成，能量带亮起。
   - **交互**：散热风扇开始根据 Token 速率进行变速旋转。机器人可以在基座上进行轻微的小幅度弹跳。
4. **Stage 4 (Walker · 武装完全体)**
   * **形态**：机械爪拼装上线，底部长出双腿或滚动轮。
   - **交互**：支持完整的物理抓取、原地转圈及行走动画。

### 待讨论的规划议题 (Open Questions)
* **状态持久化**：Zibbo 的升级数据、本地累积 Token 及组装外观应保存在 `localStorage`，还是与本地的配置文件（如 `.zibborc`）双向同步？
* **背景噪音 (Hum)**：是否需要引入极低音量的“机甲空转白噪音”作为开发者的环境背景伴奏音？
